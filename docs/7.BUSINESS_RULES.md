### 1.2 Aturan Dasar Role

**BR-ROLE-01:** Setiap pengguna sistem memiliki tepat satu role: `admin` atau `cashier`.

**BR-ROLE-02:** Role tidak dapat diubah oleh pengguna itu sendiri — hanya bisa diubah oleh Admin lain.

**BR-ROLE-03:** Akun pertama yang dibuat di sistem secara otomatis mendapat role `admin`.

**BR-ROLE-04:** Sistem harus selalu memiliki minimal satu akun Admin yang aktif. Penghapusan Admin terakhir tidak diizinkan.

**BR-ROLE-05:** Kasir yang diredirect ke URL yang bukan haknya akan diarahkan ulang ke halaman POS.

## 2. Aturan Transaksi POS

### 2.1 Aturan Validasi Sebelum Transaksi

**BR-POS-01:** Transaksi tidak dapat diproses jika keranjang kosong.

**BR-POS-02:** Transaksi tidak dapat diproses jika stok salah satu produk dalam keranjang = 0.

**BR-POS-03:** Kuantitas produk dalam satu item keranjang tidak boleh melebihi stok yang tersedia.

**BR-POS-04:** Channel penjualan wajib dipilih sebelum transaksi diproses (tidak ada default otomatis).

**BR-POS-05:** Nilai total transaksi tidak boleh nol atau negatif.

### 2.2 Aturan Pemrosesan Transaksi

**BR-POS-06:** Pembuatan transaksi bersifat **atomik** — semua INSERT (transaksi + detail) dan UPDATE (stok) terjadi dalam satu database transaction. Jika salah satu gagal, semua dibatalkan (rollback).

**BR-POS-07:** Harga satuan (unit_price) dan harga pokok (unit_cost) di `transaction_details` adalah **snapshot saat transaksi**, bukan referensi ke harga produk saat ini. Ini memastikan laporan historis akurat meski harga berubah di masa depan.

**BR-POS-08:** Setiap transaksi secara otomatis mencatat: timestamp, ID kasir, dan channel penjualan.

**BR-POS-09:** Stok produk berkurang secara otomatis setelah transaksi berhasil dikonfirmasi.

### 2.3 Aturan Pembatalan Transaksi

**BR-POS-10:** Kasir dapat membatalkan transaksi miliknya sendiri dalam **60 menit** setelah dibuat.

**BR-POS-11:** Pembatalan transaksi setelah 60 menit memerlukan **approval Admin**.

**BR-POS-12:** Setiap pembatalan transaksi harus menyertakan **alasan pembatalan** (mandatory text field).

**BR-POS-13:** Pembatalan transaksi akan **mengembalikan stok** produk ke nilai sebelum transaksi.

**BR-POS-14:** Transaksi yang sudah dibatalkan tidak dapat dibatalkan lagi atau dipulihkan.

### 2.4 Threshold Transaksi

| Threshold | Aksi Sistem |
|---|---|
| Total > Rp 500.000 | Konfirmasi ulang sebelum proses ("Apakah Anda yakin?") |
| Total > Rp 5.000.000 | Wajib isi nama pelanggan |
| Total > Rp 50.000.000 | Wajib isi nama + telepon pelanggan + flag AML |
| Diskon > 30% dari total | Memerlukan approval Admin |

## 3. Aturan Manajemen Inventaris

### 3.1 Aturan SKU

**BR-INV-01:** Format SKU: `[KATEGORI]-[NAMA_SINGKAT]-[UKURAN]-[WARNA_KODE]`  
Contoh: `GM-ALZ-XL-RBL` (Gamis, Al-Azhar, XL, Royal Blue)

**BR-INV-02:** SKU bersifat **unik** di seluruh sistem. Duplikasi SKU tidak diizinkan.

**BR-INV-03:** SKU hanya boleh mengandung huruf kapital, angka, dan tanda hubung (`-`).

**BR-INV-04:** SKU tidak dapat diubah setelah produk memiliki riwayat transaksi.

### 3.2 Aturan Harga

**BR-INV-05:** Harga jual (sell_price) harus **≥ harga pokok (cost_price)** — sistem memperingatkan (bukan memblokir) jika margin negatif.

**BR-INV-06:** Perubahan harga jual > 20% dari harga sebelumnya memerlukan **approval Admin lain** (four-eyes).

**BR-INV-07:** Harga tidak dapat bernilai negatif atau nol.

### 3.3 Aturan Stok

**BR-INV-08:** Stok tidak dapat bernilai negatif. Sistem memblokir update stok jika hasilnya < 0.

**BR-INV-09:** Ambang batas stok rendah (low_stock_threshold) default adalah **5 unit**, dapat dikonfigurasi per produk oleh Admin.

**BR-INV-10:** Produk dengan stok = 0 ditandai secara visual dengan badge "Habis" di katalog POS dan daftar inventaris.

**BR-INV-11:** Restock ≤ 100 unit dapat dilakukan langsung oleh Admin.

**BR-INV-12:** Restock > 100 unit dalam satu aksi memerlukan **approval Admin lain**.

### 3.4 Aturan Hapus Produk

**BR-INV-13:** Produk tidak dapat **dihapus secara permanen** jika memiliki riwayat transaksi — hanya dapat **dinonaktifkan** (soft delete: `is_active = false`).

**BR-INV-14:** Penghapusan/penonaktifan produk selalu memerlukan **approval Admin lain** (four-eyes).

**BR-INV-15:** Produk yang dinonaktifkan tidak muncul di katalog POS namun tetap ada di laporan historis.

## 4. Approval Workflow

### 4.1 Jenis Aksi yang Memerlukan Approval

| Kode | Aksi | Requester | Approver Minimum |
|---|---|---|---|
| APR-01 | Restock massal > 100 unit | Admin | 1 Admin lain |
| APR-02 | Hapus/nonaktifkan produk | Admin | 1 Admin lain |
| APR-03 | Batal transaksi > 60 menit | Admin | 1 Admin lain |
| APR-04 | Ubah harga > 20% | Admin | 1 Admin lain |
| APR-05 | Nonaktifkan akun pengguna | Admin | 1 Admin lain |
| APR-06 | Export data CSV | Admin | 1 Admin lain |
| APR-07 | Diskon > 30% pada transaksi | Kasir/Admin | Admin |

### 4.2 Alur Approval

```
Requester                   Sistem                  Approver
    │                          │                        │
    │── Ajukan aksi ──────────►│                        │
    │   + alasan/keterangan    │── Buat approval req ──►│
    │                          │── Notifikasi WA/app ──►│
    │                          │                        │
    │                          │◄── Approve/Reject ─────│
    │                          │   + catatan (opsional) │
    │                          │                        │
    │                          │── Log ke audit_trail   │
    │◄── Notifikasi hasil ─────│                        │
    │                          │                        │
    │                          │── [Jika Approved]      │
    │                          │── Eksekusi aksi        │
    │◄── Konfirmasi eksekusi ──│                        │
```

### 4.3 Aturan Approval

**BR-APR-01:** Requester tidak dapat menyetujui request-nya sendiri.

**BR-APR-02:** Approval request yang tidak ditanggapi dalam **24 jam** otomatis expire dengan status `expired`.

**BR-APR-03:** Aksi yang sudah di-approve tidak dapat di-reject setelah dieksekusi.

**BR-APR-04:** Jika Approver menolak, Requester menerima notifikasi beserta alasan penolakan.

**BR-APR-05:** Seluruh histori approval (approved/rejected/expired) tersimpan permanen di sistem.

## 5. Alur Program (Business Process)

### 5.1 Alur Onboarding Pengguna Baru

```
1. Admin membuat akun baru (email + nama + role)
2. Sistem mengirim email undangan via Supabase Auth
3. Pengguna mengklik link undangan → set password baru
4. Login pertama → redirect sesuai role (Admin→Dashboard, Kasir→POS)
5. Admin mewajibkan pengguna mengubah password saat login pertama
```

### 5.2 Alur Kasir Harian

```
Mulai shift:
1. Kasir login dengan kredensial
2. Diarahkan ke halaman POS
3. Verifikasi stok tersedia secara visual

Proses transaksi:
4. Cari produk (nama/SKU)
5. Pilih varian (ukuran/warna jika tersedia beberapa)
6. Set kuantitas
7. Pilih channel penjualan
8. Tinjau keranjang + total
9. Konfirmasi transaksi → stok berkurang otomatis

Akhir shift:
10. Kasir logout
11. Admin meninjau laporan hari itu via Dashboard
```

### 5.3 Alur Admin Harian

```
Pagi:
1. Login → lihat Dashboard (KPI hari sebelumnya)
2. Cek produk stok rendah → putuskan restock
3. Tinjau approval requests yang pending

Siang/sore:
4. Monitor transaksi real-time jika diperlukan
5. Proses restock dari supplier (input manual atau via OCR nota)

Malam:
6. Terima laporan harian via WhatsApp (19:00 WIB)
7. Tinjau summary revenue & profit
8. Approval transaksi mencurigakan (jika ada)
```

### 5.4 Alur Restock Produk

```
Admin                         Sistem                        Admin Approver
  │                              │                               │
  │── Input: SKU + qty restock ─►│                               │
  │                              │── [Jika qty ≤ 100]            │
  │                              │── Update stok langsung        │
  │◄── Sukses + audit log ───────│                               │
  │                              │                               │
  │── Input: SKU + qty > 100 ───►│                               │
  │                              │── Buat approval request ─────►│
  │◄── "Menunggu approval" ──────│── Notifikasi WA ─────────────►│
  │                              │                               │
  │                              │◄── Approve ───────────────────│
  │                              │── Update stok                 │
  │◄── Notifikasi: stok diupdate │                               │
```

## 6. Alur Verifikasi Masjid/Lembaga Zakat

### 6.1 Prasyarat

Modul Zakat hanya dapat diakses oleh Admin. Alokasi zakat dihitung dari profit yang diinput secara manual oleh Admin.

### 6.2 Alur Pendaftaran dan Verifikasi Lembaga

```
Langkah 1 — Admin Mendaftarkan Lembaga:
  - Input: nama, tipe (masjid/yayasan/lembaga), alamat, kontak
  - Input: data rekening bank (nama bank, nomor rekening, nama pemilik rekening)
  - Upload dokumen pendukung (SK, foto, dll.)
  - Status otomatis: "pending"

Langkah 2 — Verifikasi Oleh Admin Lain (Four-Eyes):
  - Admin kedua meninjau data dan dokumen yang diupload
  - Validasi: rekening bank atas nama lembaga (bukan pribadi)?
  - Validasi: nomor SK/dokumen resmi valid?
  - Keputusan: Verifikasi ("verified") atau Tolak ("rejected") dengan alasan

Langkah 3 — Penyaluran Dana:
  - Hanya lembaga berstatus "verified" yang muncul di pilihan penerima zakat
  - Admin menginput jumlah dana yang dialokasikan
  - Sistem mencatat tanggal, jumlah, dan lembaga penerima
  - Admin mengkonfirmasi transfer manual telah dilakukan
  - Sistem mencatat status transfer: "transferred"
```

### 6.3 Aturan Zakat

**BR-ZKT-01:** Dana hanya dapat disalurkan ke lembaga berstatus `verified`.

**BR-ZKT-02:** Setiap penyaluran dana dicatat dalam tabel `zakat_distributions` dengan detail lengkap.

**BR-ZKT-03:** Verifikasi lembaga memerlukan approval dari Admin yang berbeda dengan yang mendaftarkan (four-eyes).

**BR-ZKT-04:** Lembaga yang sudah diverifikasi dapat ditangguhkan (`suspended`) jika ada indikasi masalah, tanpa menghapus histori penyaluran sebelumnya.

## 7. Aturan Kalkulasi Keuangan

### 7.1 Formula Standar

```
Subtotal per item   = quantity × unit_price
Total Revenue       = Σ (subtotal semua item dalam transaksi)
Total Cost          = Σ (quantity × unit_cost semua item)
Total Profit        = Total Revenue - Total Cost
Margin %            = (Total Profit / Total Revenue) × 100

Profit per item     = quantity × (unit_price - unit_cost)
```

### 7.2 Aturan Pembulatan

**BR-FIN-01:** Semua nilai mata uang disimpan dalam **DECIMAL(12,2)** (akurasi hingga sen).

**BR-FIN-02:** Tampilan di UI menggunakan format **Rupiah tanpa desimal** (dibulatkan ke satuan terdekat) karena mata uang Rupiah tidak menggunakan sen dalam praktik sehari-hari.

**BR-FIN-03:** Pembulatan diterapkan **hanya saat tampilan UI**, bukan saat penyimpanan di database.

### 7.3 Aturan Diskon

**BR-FIN-04:** Diskon diterapkan pada level **total transaksi** (bukan per item).

**BR-FIN-05:** Diskon maksimum adalah **50%** dari total transaksi.

**BR-FIN-06:** Diskon > 30% memerlukan approval Admin sebelum transaksi dapat diproses.

## 8. Aturan Notifikasi

### 8.1 Trigger Notifikasi

| Event | Penerima | Channel | Waktu |
|---|---|---|---|
| Stok produk < threshold | Admin | WhatsApp + in-app | Real-time |
| Transaksi > Rp 1.000.000 | Admin | WhatsApp | Real-time |
| Approval request baru | Approver | WhatsApp + in-app | Real-time |
| Approval disetujui/ditolak | Requester | In-app | Real-time |
| Login dari IP baru | User terkait | Email | Segera |
| Laporan harian | Admin | WhatsApp | 19:00 WIB |
| Laporan mingguan | Admin | Email | Senin 08:00 WIB |

### 8.2 Aturan Notifikasi

**BR-NOT-01:** Notifikasi stok rendah dikirim **sekali per produk per hari** — tidak berulang setiap jam.

**BR-NOT-02:** Admin dapat mengkonfigurasi jam "do not disturb" (default: 22:00–07:00 WIB) untuk notifikasi WhatsApp non-kritis.

**BR-NOT-03:** Notifikasi keamanan (login baru, aksi kritis) **selalu dikirim** tanpa memperhatikan jam.
