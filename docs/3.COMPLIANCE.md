## 1. KYC & AML

### 1.1 Konteks Aplikasi

Maheer Fashion POS adalah sistem POS retail fashion dengan nilai transaksi rata-rata di bawah ambang batas pelaporan OJK/PPATK. Namun, sebagai praktik tata kelola yang baik, sistem menerapkan langkah-langkah mitigasi risiko berikut:

### 1.2 Identifikasi Pelanggan (KYC Dasar)

| Level | Kewajiban | Threshold |
|---|---|---|
| **Transaksi Biasa** | Nama pelanggan (opsional) | < Rp 5.000.000 |
| **Transaksi Besar** | Nama + nomor telepon wajib dicatat | Rp 5.000.000 – Rp 50.000.000 |
| **Transaksi Sangat Besar** | Nama + telepon + ID pelanggan + catatan transaksi wajib | > Rp 50.000.000 |

> **Catatan:** Batas Rp 50 juta mengacu pada Peraturan Menteri Keuangan (PMK) tentang kewajiban pelaporan transaksi tunai. Tim Legal wajib memverifikasi threshold terbaru sebelum go-live.

### 1.3 AML Red Flags

Sistem akan menandai (flag) transaksi yang memenuhi kriteria berikut untuk ditinjau oleh Admin:

- Transaksi tunai di atas Rp 50.000.000 dalam satu hari dari satu pelanggan.
- Pola transaksi berulang dalam jumlah kecil dari pelanggan yang sama dalam periode singkat (structuring).
- Transaksi di luar jam operasional normal (dapat dikonfigurasi Admin).

### 1.4 Implementasi Teknis

```typescript
// Flag transaksi mencurigakan sebelum disimpan
async function checkAMLFlags(transactionData: TransactionInput): Promise<AMLCheckResult> {
  const flags: string[] = [];

  // Check 1: Nilai tunggal sangat besar
  if (transactionData.total_revenue > 50_000_000) {
    flags.push('HIGH_VALUE_TRANSACTION');
  }

  // Check 2: Frekuensi transaksi dari customer yang sama
  const recentTxns = await getRecentTransactionsByCustomer(
    transactionData.customer_name,
    hoursBack: 24
  );
  if (recentTxns.length > 10) {
    flags.push('HIGH_FREQUENCY_CUSTOMER');
  }

  return { hasFlags: flags.length > 0, flags };
}
```

## 2. Audit Trail Specification

### 2.1 Prinsip Audit Trail

Setiap perubahan data sensitif di sistem harus dapat ditelusuri (immutable log). Audit trail **tidak boleh dihapus** — hanya dapat diarsipkan.

### 2.2 Tabel `audit_logs`

```sql
CREATE TABLE audit_logs (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  timestamp     TIMESTAMPTZ NOT NULL DEFAULT now(),
  actor_id      UUID NOT NULL,         -- ID pengguna yang melakukan aksi
  actor_role    VARCHAR(20) NOT NULL,  -- Role saat aksi dilakukan
  action        VARCHAR(50) NOT NULL,  -- CREATE, UPDATE, DELETE, LOGIN, LOGOUT
  resource_type VARCHAR(50) NOT NULL,  -- 'product', 'transaction', 'user'
  resource_id   UUID,                  -- ID entitas yang dimodifikasi
  old_value     JSONB,                 -- Nilai sebelum perubahan
  new_value     JSONB,                 -- Nilai setelah perubahan
  ip_address    INET,
  user_agent    TEXT,
  metadata      JSONB                  -- Data tambahan kontekstual
);

-- Audit log tidak boleh diupdate atau didelete
CREATE RULE no_update_audit_logs AS ON UPDATE TO audit_logs DO INSTEAD NOTHING;
CREATE RULE no_delete_audit_logs AS ON DELETE TO audit_logs DO INSTEAD NOTHING;
```

### 2.3 Event yang Wajib Di-audit

| Event | Resource | Data yang Dicatat |
|---|---|---|
| Login berhasil | `user` | timestamp, ip_address, user_agent |
| Login gagal | `user` | timestamp, ip_address, alasan gagal |
| Logout | `user` | timestamp, durasi sesi |
| Tambah produk | `product` | seluruh data produk baru |
| Edit produk | `product` | field yang berubah (old vs new value) |
| Hapus/nonaktifkan produk | `product` | data produk sebelum dinonaktifkan |
| Restock | `product` | stok lama, jumlah ditambah, stok baru |
| Buat transaksi | `transaction` | seluruh data transaksi + detail item |
| Batal transaksi | `transaction` | data transaksi, alasan pembatalan |
| Buat user baru | `user` | data user baru (email, role) |
| Ubah role user | `user` | role lama, role baru, actor |
| Nonaktifkan user | `user` | data user, actor |
| Akses laporan keuangan | `report` | timestamp, filter yang digunakan |

### 2.4 Implementasi via Supabase Trigger

```sql
-- Trigger otomatis saat produk diupdate
CREATE OR REPLACE FUNCTION log_product_changes()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO audit_logs (
    actor_id, actor_role, action, resource_type, resource_id, old_value, new_value
  )
  VALUES (
    auth.uid(),
    (SELECT role FROM users WHERE id = auth.uid()),
    TG_OP,
    'product',
    NEW.id,
    to_jsonb(OLD),
    to_jsonb(NEW)
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER audit_product_changes
  AFTER INSERT OR UPDATE OR DELETE ON products
  FOR EACH ROW EXECUTE FUNCTION log_product_changes();
```

## 3. Four-Eyes Rules (Prinsip Empat Mata)

### 3.1 Definisi

Prinsip empat mata mengharuskan **dua orang berbeda** untuk menyetujui aksi kritis sebelum dieksekusi. Ini mencegah fraud internal dan kesalahan fatal oleh satu orang.

### 3.2 Aksi yang Memerlukan Four-Eyes Approval

| Aksi | Threshold | Approver |
|---|---|---|
| Restock massal (> 100 unit sekaligus) | > 100 unit | Admin lain |
| Hapus permanen produk | Semua kasus | Admin lain |
| Pembatalan transaksi setelah 1 jam | Semua kasus | Admin |
| Ubah harga produk > 20% | Perubahan > 20% | Admin lain |
| Nonaktifkan akun pengguna | Semua kasus | Admin lain |
| Akses export data lengkap | Semua kasus | Admin lain |

### 3.3 Alur Approval

```
Requester (Admin A)          System              Approver (Admin B)
      │                        │                       │
      │── Ajukan aksi kritis ─►│                       │
      │                        │── Buat approval_request│
      │                        │── Kirim notifikasi ───►│
      │                        │                       │
      │                        │                       │── Review detail
      │                        │                       │── Approve / Reject
      │                        │◄── Keputusan ─────────│
      │                        │                       │
      │                        │── [Jika Approve]       │
      │                        │── Eksekusi aksi        │
      │                        │── Log ke audit_trail   │
      │◄── Notifikasi hasil ───│                       │
```

### 3.4 Tabel `approval_requests`

```sql
CREATE TABLE approval_requests (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  requester_id    UUID NOT NULL REFERENCES users(id),
  approver_id     UUID REFERENCES users(id),  -- NULL sampai ada yang approve
  action_type     VARCHAR(100) NOT NULL,
  resource_type   VARCHAR(50) NOT NULL,
  resource_id     UUID,
  request_data    JSONB NOT NULL,   -- Data lengkap aksi yang diajukan
  status          VARCHAR(20) NOT NULL DEFAULT 'pending',
                  -- 'pending', 'approved', 'rejected', 'expired'
  decided_at      TIMESTAMPTZ,
  rejection_reason TEXT,
  expires_at      TIMESTAMPTZ NOT NULL  -- Auto-expire setelah 24 jam
);
```

### 3.5 Aturan Tambahan

- Requester tidak boleh menjadi approver untuk request-nya sendiri (enforced di RLS).
- Request yang tidak disetujui dalam 24 jam otomatis expire dan tidak dapat dieksekusi.
- Semua keputusan approval tercatat di `audit_logs`.

## 4. Mosque/Lembaga Verification (Fitur Zakat)

### 4.1 Konteks

Maheer Fashion POS memiliki modul **Zakat** yang memungkinkan sebagian keuntungan disalurkan ke lembaga/masjid terverifikasi. Verifikasi diperlukan untuk memastikan dana tersalur ke lembaga yang sah.

### 4.2 Status Verifikasi Lembaga

| Status | Deskripsi | Aksi yang Diizinkan |
|---|---|---|
| `pending` | Baru didaftarkan, belum diverifikasi | Tidak bisa menerima dana |
| `verified` | Telah diverifikasi Admin | Dapat menerima dana Zakat |
| `rejected` | Ditolak karena data tidak valid | Tidak bisa menerima dana |
| `suspended` | Dibekukan sementara | Tidak bisa menerima dana |

### 4.3 Dokumen Verifikasi yang Diperlukan

Lembaga/masjid yang ingin terdaftar harus menyediakan:

1. **Nama lengkap lembaga** sesuai akta/SK pendirian.
2. **Nomor SK Kemenkumham** (untuk yayasan) atau **Surat Keterangan Domisili** (untuk masjid).
3. **Nomor rekening bank** atas nama lembaga (bukan rekening pribadi).
4. **Nama dan kontak penanggung jawab**.
5. **Foto bangunan/lokasi** (opsional, untuk konfirmasi keberadaan fisik).

### 4.4 Alur Verifikasi

```
Pendaftaran Lembaga       Admin Maheer          Penyaluran Dana
       │                      │                       │
       │── Submit data ───────►│                       │
       │   + dokumen           │                       │
       │                      │── Review dokumen       │
       │                      │── Verifikasi manual    │
       │                      │                       │
       │                      │── [Jika Valid]         │
       │                      │── Status → 'verified'  │
       │◄── Notifikasi ────────│                       │
       │    terverifikasi      │                       │
       │                      │                       │── Dana dapat disalurkan
```

### 4.5 Tabel `zakat_recipients`

```sql
CREATE TABLE zakat_recipients (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name             VARCHAR(200) NOT NULL,
  type             VARCHAR(50) NOT NULL,  -- 'masjid', 'yayasan', 'lembaga'
  address          TEXT,
  contact_person   VARCHAR(100),
  contact_phone    VARCHAR(20),
  bank_name        VARCHAR(100),
  bank_account_no  VARCHAR(50),
  bank_account_name VARCHAR(100),
  verification_status VARCHAR(20) NOT NULL DEFAULT 'pending',
  verified_by      UUID REFERENCES users(id),
  verified_at      TIMESTAMPTZ,
  rejection_reason TEXT,
  documents        JSONB,  -- Array URL dokumen yang diupload
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 5. Privacy Policy

### 5.1 Data Pribadi yang Dikumpulkan

| Data | Sumber | Tujuan |
|---|---|---|
| Alamat email staf | Pendaftaran akun | Autentikasi login |
| Nama lengkap staf | Pendaftaran akun | Identifikasi di log transaksi |
| Nama pelanggan | Input kasir (opsional) | Histori pembelian pelanggan |
| Nomor telepon pelanggan | Input kasir (untuk transaksi besar) | Kepatuhan AML |
| Alamat IP | Otomatis dari server | Audit keamanan |
| User-agent browser | Otomatis dari server | Audit keamanan |

### 5.2 Data yang TIDAK Dikumpulkan

- Nomor KTP atau identitas pemerintah pelanggan (kecuali transaksi > Rp 50 juta).
- Data biometrik.
- Data lokasi GPS perangkat.
- Password dalam bentuk plaintext (selalu di-hash oleh Supabase Auth).

### 5.3 Dasar Hukum Pemrosesan Data

Pemrosesan data pribadi dilakukan berdasarkan:
- **UU No. 27 Tahun 2022** tentang Pelindungan Data Pribadi (UU PDP Indonesia).
- **Kepentingan sah bisnis** untuk pengelolaan operasional toko.
- **Kepatuhan regulasi** untuk data yang diwajibkan oleh hukum (AML).

### 5.4 Hak Subjek Data

Staf dan pelanggan memiliki hak:
- **Hak Akses:** Meminta salinan data pribadi yang tersimpan.
- **Hak Koreksi:** Meminta perbaikan data yang tidak akurat.
- **Hak Penghapusan:** Meminta penghapusan data (dengan batasan hukum — data transaksi wajib disimpan minimal 5 tahun).
- **Hak Pembatasan:** Meminta pembatasan pemrosesan dalam kondisi tertentu.

### 5.5 Pengungkapan kepada Pihak Ketiga

Data pribadi hanya diungkapkan kepada:
- **Supabase (PostgreSQL hosting):** Sebagai sub-prosesor data dengan standar keamanan SOC 2 Type II.
- **Vercel (hosting aplikasi):** Hanya menerima request HTTP, tidak menyimpan data pengguna.
- **Otoritas berwenang:** Jika diwajibkan oleh hukum (contoh: PPATK untuk laporan AML).

## 6. Data Retention Policy

### 6.1 Tabel Retensi Data

| Jenis Data | Periode Retensi | Dasar | Aksi Setelah Kadaluarsa |
|---|---|---|---|
| Data transaksi penjualan | **5 tahun** | Hukum Pajak & Perdagangan Indonesia | Diarsipkan ke cold storage |
| Audit logs | **5 tahun** | Kepatuhan operasional | Diarsipkan, tidak dihapus |
| Data akun staf (aktif) | Selama aktif | Operasional | — |
| Data akun staf (nonaktif) | **1 tahun** setelah nonaktif | Grace period reaktivasi | Anonymized / dihapus |
| Data pelanggan (opsional) | **3 tahun** tanpa transaksi | Bisnis | Dihapus otomatis |
| Log autentikasi | **1 tahun** | Keamanan | Dihapus otomatis |
| Approval requests | **3 tahun** | Audit internal | Diarsipkan |
| Data Zakat | **5 tahun** | Pertanggungjawaban syariah & pajak | Diarsipkan |

### 6.2 Proses Penghapusan & Anonimisasi

```sql
-- Contoh: Anonimisasi data pelanggan setelah 3 tahun tidak bertransaksi
UPDATE transactions
SET customer_name = 'ANONYMIZED'
WHERE customer_name IS NOT NULL
  AND created_at < NOW() - INTERVAL '3 years'
  AND id NOT IN (
    SELECT DISTINCT transaction_id FROM transaction_details
    WHERE created_at >= NOW() - INTERVAL '3 years'
  );
```

### 6.3 Backup & Cold Storage

- **Backup harian** dilakukan otomatis oleh Supabase (retensi 7 hari untuk plan gratis, 30 hari untuk Pro).
- Data yang telah melewati periode retensi aktif dieksport ke **cold storage** (Amazon S3 Glacier atau Google Cloud Archive) sebelum dihapus dari database utama.
- Jadwal review retensi: **setiap tahun** oleh tim Developer + Legal.

### 6.4 Pemusnahan Data

Pemusnahan data dilakukan dengan:
- **Database:** `DELETE` atau `UPDATE` (anonimisasi) dengan audit log.
- **Backup files:** Secure deletion menggunakan perintah yang memastikan data tidak dapat dipulihkan.
- **Dokumen fisik** (jika ada): Shredding fisik dan dicatat dalam berita acara pemusnahan.
