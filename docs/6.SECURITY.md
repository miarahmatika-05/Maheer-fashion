## 1. Security Architecture

### 1.1 Defense in Depth

Keamanan aplikasi diimplementasikan dalam beberapa lapisan:

```
Layer 1 — Network:     HTTPS/TLS 1.3, Vercel Edge (DDoS protection)
Layer 2 — Auth:        Supabase Auth (JWT RS256, refresh token rotation)
Layer 3 — Application: Role-based routing, input validation, CSRF protection
Layer 4 — Database:    Row Level Security (RLS), parameterized queries
Layer 5 — Audit:       Immutable audit logs, session tracking
```

### 1.2 Autentikasi & Sesi

| Aspek | Implementasi |
|---|---|
| Mekanisme login | Email + Password via Supabase Auth |
| Token type | JWT (RS256) |
| Access token TTL | 1 jam |
| Refresh token TTL | 7 hari |
| Storage | HttpOnly cookie (SSR) — bukan localStorage |
| Auto-logout | Setelah 30 menit tidak aktif |
| Session invalidation | Logout di semua device saat password diubah |

### 1.3 Password Policy

- Minimal **8 karakter**.
- Mengandung setidaknya **1 huruf besar, 1 huruf kecil, 1 angka**.
- Tidak boleh sama dengan 3 password terakhir.
- Admin wajib mengatur ulang password kasir baru saat onboarding pertama.

### 1.4 Row Level Security (RLS) — Overview

| Tabel | Admin | Kasir |
|---|---|---|
| `users` | SELECT, INSERT, UPDATE | SELECT (diri sendiri) |
| `products` | ALL | SELECT (is_active=true) |
| `transactions` | ALL | SELECT + INSERT (milik sendiri) |
| `transaction_details` | ALL | SELECT + INSERT (milik sendiri) |
| `audit_logs` | SELECT | — |
| `approval_requests` | ALL | — |

### 1.5 Input Validation

Semua input dari user divalidasi di dua level:

**Frontend (UX validation):**

```typescript
import { z } from 'zod';

const ProductSchema = z.object({
  sku: z.string().regex(/^[A-Z]{2}-[A-Z]{3}-[A-Z0-9]+-[A-Z]{3}$/, 'Format SKU tidak valid'),
  name: z.string().min(2).max(200),
  sellPrice: z.number().min(0).max(999_999_999),
  costPrice: z.number().min(0).max(999_999_999),
  currentStock: z.number().int().min(0),
});
```

**Backend (Server Action/API):** Validasi ulang dengan schema yang sama sebelum query database.

### 1.6 Secrets Management

| Secret | Storage | Rotasi |
|---|---|---|
| `SUPABASE_SERVICE_ROLE_KEY` | Vercel Environment Variables | Setiap 90 hari |
| `GOOGLE_VISION_API_KEY` | Vercel Environment Variables | Setiap 90 hari |
| `WA_ACCESS_TOKEN` | Vercel Environment Variables | Setiap 60 hari |
| `NEXTAUTH_SECRET` | Vercel Environment Variables | Setiap 180 hari |
| JWT signing keys | Dikelola Supabase (auto-rotate) | Otomatis |

**Aturan:**
- Secrets **tidak pernah** di-commit ke repository, bahkan di branch private.
- Setiap secret yang diduga bocor harus **segera dirotasi** dan insiden dilaporkan.
- Gunakan Vercel Environment Variables UI untuk menyimpan secrets — bukan file `.env` di server.

## 2. Incident Response

### 2.1 Klasifikasi Insiden

| Severity | Deskripsi | Contoh | SLA Response |
|---|---|---|---|
| **P1 — Critical** | Layanan tidak dapat diakses atau data bocor | Database tidak bisa diakses, credential bocor | 15 menit |
| **P2 — High** | Fungsi utama tidak berfungsi | POS tidak bisa proses transaksi | 1 jam |
| **P3 — Medium** | Fungsi minor terganggu | Dashboard chart tidak muncul | 4 jam |
| **P4 — Low** | Estetika atau UX minor | Teks salah ejaan | 1 hari kerja |

### 2.2 Incident Response Playbook

#### Langkah 1: Deteksi & Triase (0–15 menit untuk P1)

```
1. Terima alert dari monitoring (Vercel, Supabase, atau laporan user)
2. Akses dashboard monitoring dan identifikasi scope
3. Klasifikasikan severity (P1–P4)
4. Notifikasi tim sesuai severity:
   - P1: Telepon langsung + WhatsApp group "Maheer Tech Emergency"
   - P2-P3: WhatsApp group "Maheer Tech"
   - P4: Ticket di issue tracker
```

#### Langkah 2: Containment (untuk P1 & P2)

```
P1 — Data breach:
  1. Segera revoke semua active sessions: Supabase Auth → Revoke all tokens
  2. Nonaktifkan API keys yang diduga compromised
  3. Aktifkan maintenance mode (deploy halaman statis)
  4. Isolasi akun user yang terdampak

P1 — Service down:
  1. Cek status Supabase: status.supabase.com
  2. Cek status Vercel: vercel-status.com
  3. Jika provider bermasalah: tunggu dan update status ke stakeholder
  4. Jika kode bermasalah: rollback ke deployment terakhir yang stabil
```

#### Langkah 3: Eradication & Recovery

```
1. Identifikasi root cause dari logs dan audit trail
2. Terapkan perbaikan (hotfix branch)
3. Uji di staging environment
4. Deploy dengan approval dua orang (four-eyes)
5. Monitor 30 menit setelah deployment
```

#### Langkah 4: Post-Incident Review (dalam 48 jam)

Template laporan:

```
## Incident Report — [Tanggal]

**Severity:** P[1-4]
**Duration:** [Waktu mulai] – [Waktu selesai]
**Impact:** [Berapa user terdampak? Fungsi apa?]

### Timeline
- HH:MM — [Event]
- HH:MM — [Event]

### Root Cause
[Penjelasan teknis penyebab insiden]

### Resolution
[Langkah yang diambil untuk menyelesaikan]

### Lessons Learned
[Apa yang bisa diperbaiki ke depannya?]

### Action Items
- [ ] [Tindakan perbaikan 1] — PIC: [nama] — Deadline: [tanggal]
```

## 3. Backup & Recovery

### 3.1 Backup Strategy

| Data | Frekuensi | Retensi | Provider |
|---|---|---|---|
| Database PostgreSQL (full) | Setiap hari (02:00 WIB) | 30 hari (Supabase Pro) | Supabase |
| Database PostgreSQL (WAL) | Continuous (Point-in-Time Recovery) | 7 hari | Supabase |
| Storage files (dokumen OCR) | Setiap hari (03:00 WIB) | 90 hari | Supabase Storage |
| Kode aplikasi | Setiap commit | Selamanya | GitHub |
| Environment variables | Manual (setiap perubahan) | Dokumen terenkripsi | 1Password / Bitwarden |

### 3.2 Recovery Time Objective (RTO) & Recovery Point Objective (RPO)

| Skenario | RTO Target | RPO Target |
|---|---|---|
| Rollback deployment | 5 menit | N/A (kode) |
| Restore database dari backup harian | 2 jam | 24 jam |
| Point-in-Time Recovery (PITR) | 30 menit | 5 menit |
| Full disaster recovery | 4 jam | 1 jam |

### 3.3 Prosedur Restore Database

```bash
# 1. Akses Supabase Dashboard → Settings → Database → Backups
# 2. Pilih backup point yang diinginkan
# 3. Klik "Restore" — konfirmasi bahwa ini akan OVERWRITE database saat ini

# Atau via Supabase CLI untuk PITR:
supabase db restore --target-time "2026-05-01T10:00:00Z"

# Verifikasi setelah restore:
# - Periksa jumlah record di tabel utama
# - Jalankan smoke test via Playwright
# - Konfirmasi ke stakeholder
```

### 3.4 Disaster Recovery Drill

Drill dilakukan **setiap 6 bulan** untuk memverifikasi bahwa prosedur restore berfungsi:

1. Jadwalkan jendela waktu maintenance (di luar jam kerja).
2. Restore database ke environment staging dari backup terbaru.
3. Jalankan E2E test suite di staging.
4. Dokumentasikan waktu yang dibutuhkan dan temuan.
5. Update runbook jika ada perubahan prosedur.

## 4. Monitoring & Alerting

### 4.1 Health Check Endpoint

```typescript
// app/api/health/route.ts
export async function GET() {
  const checks = {
    status: 'ok',
    timestamp: new Date().toISOString(),
    database: 'unknown',
    version: process.env.npm_package_version,
  };

  try {
    const supabase = createServerClient();
    await supabase.from('users').select('id').limit(1);
    checks.database = 'ok';
  } catch {
    checks.database = 'error';
    checks.status = 'degraded';
  }

  const statusCode = checks.status === 'ok' ? 200 : 503;
  return Response.json(checks, { status: statusCode });
}
```

### 4.2 Metrics yang Dipantau

| Metrik | Tool | Threshold Alert |
|---|---|---|
| Uptime aplikasi | Vercel Analytics / UptimeRobot | < 99.5% dalam 1 jam |
| Response time API | Vercel Analytics | > 2 detik rata-rata |
| Error rate (5xx) | Vercel Logs | > 1% dari total request |
| Database connection pool | Supabase Dashboard | > 80% pool usage |
| Storage usage | Supabase Dashboard | > 80% quota |
| Failed login attempts | Custom (audit_logs query) | > 10 gagal dari 1 IP dalam 5 menit |

### 4.3 Alerting Channels

```
P1 Alerts  → WhatsApp (langsung + telepon)
P2 Alerts  → WhatsApp group "Maheer Tech"
P3-P4 Alerts → Email + Issue tracker
Daily Summary → WhatsApp Admin pukul 07:00 WIB
```

### 4.4 Custom Alert: Failed Login Brute Force

```sql
-- Query yang dijalankan setiap 5 menit via Supabase scheduled function
SELECT ip_address, COUNT(*) as attempts
FROM audit_logs
WHERE action = 'LOGIN_FAILED'
  AND timestamp > NOW() - INTERVAL '5 minutes'
GROUP BY ip_address
HAVING COUNT(*) > 10;
-- Jika ada hasil → trigger WhatsApp alert ke Admin
```

## 5. Runbook Operasional

### RB-001: Restart Aplikasi (Vercel Redeploy)

**Kapan digunakan:** Aplikasi tidak merespons, deployment stuck.

```bash
# Via Vercel Dashboard:
# 1. Masuk ke vercel.com → Proyek maheer-fashion-pos
# 2. Deployments → Klik deployment terbaru
# 3. Klik "..." → Redeploy

# Via CLI:
vercel redeploy [deployment-url]
```

**Verifikasi:** Akses `/api/health` — pastikan status `200 OK`.

---

### RB-002: Revoke Sesi Semua Pengguna

**Kapan digunakan:** Dugaan compromise credential, kebijakan keamanan darurat.

```bash
# Via Supabase Dashboard:
# Authentication → Users → pilih user → "Sign out all sessions"

# Atau via SQL untuk semua user:
# Supabase Dashboard → SQL Editor:
DELETE FROM auth.sessions;
-- ⚠️ Semua pengguna akan ter-logout dan harus login ulang
```

---

### RB-003: Nonaktifkan Akun Kasir

**Kapan digunakan:** Kasir resign, akun diduga disalahgunakan.

```sql
-- Supabase SQL Editor:
UPDATE users
SET is_active = false
WHERE email = 'kasir@example.com';

-- Revoke session:
DELETE FROM auth.sessions
WHERE user_id = (SELECT id FROM auth.users WHERE email = 'kasir@example.com');
```

**Verifikasi:** Coba login dengan akun tersebut — harus gagal.

---

### RB-004: Manual Koreksi Stok

**Kapan digunakan:** Selisih stok ditemukan saat audit fisik.

```sql
-- WAJIB dicatat di audit_logs secara manual sebelum eksekusi
INSERT INTO audit_logs (actor_id, actor_role, action, resource_type, resource_id, old_value, new_value, metadata)
VALUES (
  '<admin_user_id>',
  'admin',
  'MANUAL_STOCK_CORRECTION',
  'product',
  '<product_id>',
  '{"current_stock": <old_value>}',
  '{"current_stock": <new_value>}',
  '{"reason": "Koreksi hasil audit fisik [tanggal]", "approved_by": "<nama>"}'
);

-- Eksekusi koreksi (wajib setelah insert log di atas)
UPDATE products
SET current_stock = <new_value>, updated_at = NOW()
WHERE id = '<product_id>';
```

### RB-005: Export Data Transaksi untuk Keperluan Pajak

**Kapan digunakan:** Pelaporan pajak, audit eksternal.

```sql
-- Export transaksi dalam rentang tanggal
COPY (
  SELECT
    t.id,
    t.created_at,
    t.channel,
    t.total_revenue,
    t.total_cost,
    t.total_profit,
    u.full_name AS cashier_name,
    t.customer_name
  FROM transactions t
  JOIN users u ON t.cashier_id = u.id
  WHERE t.created_at BETWEEN '2026-01-01' AND '2026-12-31'
  ORDER BY t.created_at
) TO '/tmp/transactions_2026.csv' WITH CSV HEADER;
```

**Catatan:** Akses export dicatat di audit_logs. Wajib mendapat approval four-eyes.
