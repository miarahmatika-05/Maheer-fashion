## 1. Coding Standards

### 1.1 Bahasa & TypeScript

- **Wajib** menggunakan TypeScript strict mode (`"strict": true` di `tsconfig.json`).
- Dilarang menggunakan `any` — gunakan `unknown` atau type yang tepat.
- Semua fungsi publik harus memiliki return type yang eksplisit.
- Gunakan `interface` untuk object shapes, `type` untuk union/intersection types.

```typescript
// ✅ Benar
interface Product {
  id: string;
  sku: string;
  name: string;
  currentStock: number;
}

async function getProductBySku(sku: string): Promise<Product | null> { ... }

// ❌ Salah
async function getProductBySku(sku: any): Promise<any> { ... }
```

### 1.2 Penamaan (Naming Conventions)

| Jenis | Konvensi | Contoh |
|---|---|---|
| Komponen React | PascalCase | `ProductCard`, `TransactionModal` |
| Fungsi / variabel | camelCase | `calculateTotal`, `isLoading` |
| Konstanta global | SCREAMING_SNAKE_CASE | `LOW_STOCK_THRESHOLD`, `MAX_RETRY_COUNT` |
| File komponen | PascalCase | `ProductCard.tsx` |
| File utilitas/hooks | camelCase | `useProducts.ts`, `formatCurrency.ts` |
| Tabel database | snake_case | `transaction_details`, `audit_logs` |
| Kolom database | snake_case | `created_at`, `cashier_id` |
| CSS classes | Tailwind utilities | `text-sm font-medium text-gray-700` |

### 1.4 Server Actions & API Routes

- **Server Actions** digunakan untuk mutasi data (insert, update, delete).
- **API Routes** digunakan untuk integrasi pihak ketiga (OCR, WhatsApp webhook).
- Semua Server Actions harus memvalidasi session sebelum eksekusi.

```typescript
// app/actions/product.ts
'use server';

import { createServerClient } from '@/lib/supabase/server';
import { revalidatePath } from 'next/cache';

export async function restockProduct(productId: string, quantity: number) {
  const supabase = createServerClient();

  // 1. Validasi session
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) throw new Error('Unauthorized');

  // 2. Validasi role
  const { data: profile } = await supabase
    .from('users')
    .select('role')
    .eq('id', user.id)
    .single();
  if (profile?.role !== 'admin') throw new Error('Forbidden');

  // 3. Eksekusi
  const { error } = await supabase.rpc('restock_product', {
    p_product_id: productId,
    p_quantity: quantity,
  });
  if (error) throw error;

  revalidatePath('/inventory');
}
```

## 2. Git Workflow

### 2.1 Branching Strategy (GitHub Flow)

```
main (production)
  └── develop (staging)
        ├── feature/POS-001-product-search
        ├── feature/INV-003-restock-form
        ├── fix/DASH-002-chart-not-loading
        └── hotfix/PROD-critical-stock-bug
```

| Branch | Tujuan | Deploy ke |
|---|---|---|
| `main` | Kode production stabil | Vercel Production |
| `develop` | Integrasi fitur baru | Vercel Staging |
| `feature/*` | Pengembangan fitur | Vercel Preview URL |
| `fix/*` | Perbaikan bug non-kritis | Vercel Preview URL |
| `hotfix/*` | Perbaikan bug kritis di production | Langsung ke `main` |

### 2.2 Konvensi Nama Branch

Format: `[type]/[ticket-id]-[short-description]`

```
feature/POS-001-cart-quantity-control
fix/INV-015-sku-validation-error
hotfix/TXN-critical-stock-not-decreasing
chore/update-supabase-client-v2
docs/add-api-spec-section
```

### 2.3 Konvensi Commit Message (Conventional Commits)

Format: `[type]([scope]): [description]`

```
feat(pos): tambah fitur pencarian produk berdasarkan SKU
fix(inventory): perbaiki validasi format SKU duplikat
feat(dashboard): tambah grafik revenue vs profit bulanan
chore(deps): upgrade Supabase JS SDK ke v2.38
docs(readme): perbarui instruksi local setup
test(pos): tambah unit test untuk kalkulasi subtotal
refactor(auth): pindahkan session handling ke middleware
```

**Types yang valid:** `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`

### 2.4 Pull Request Checklist

Setiap PR wajib memenuhi checklist berikut sebelum merge:

- [ ] Branch dari `develop` (bukan dari `main`).
- [ ] Title PR mengikuti konvensi commit message.
- [ ] Description menjelaskan: apa yang diubah, mengapa, dan screenshot jika ada perubahan UI.
- [ ] Semua unit test lulus (`npm run test`).
- [ ] Tidak ada ESLint error (`npm run lint`).
- [ ] Preview URL di Vercel sudah diperiksa secara visual.
- [ ] Minimal 1 reviewer telah approve (implementasi four-eyes untuk code review).
- [ ] Migration database sudah diuji di lingkungan staging jika ada perubahan schema.

### 2.5 Alur Merge

```
feature/xxx → (PR) → develop → (QA pass) → (PR) → main
```

- Squash merge digunakan untuk merge `feature/*` ke `develop`.
- Merge commit digunakan untuk merge `develop` ke `main` (mempertahankan histori).

## 3. Testing Strategy

### 3.1 Piramida Testing

```
        /\
       /  \
      / E2E \          ← Sedikit (Playwright) — alur kritis end-to-end
     /────────\
    / Integration \    ← Sedang — interaksi komponen + API
   /──────────────\
  /   Unit Tests   \   ← Banyak — fungsi, hooks, utilitas
 /──────────────────\
```

### 3.2 Unit Testing

**Framework:** Jest + React Testing Library

**Target coverage:** Minimum 70% pada `lib/` dan `components/ui/`.

```typescript
// __tests__/lib/utils.test.ts
import { formatCurrency, calculateCartTotal } from '@/lib/utils';

describe('formatCurrency', () => {
  it('formats number to Indonesian Rupiah', () => {
    expect(formatCurrency(185000)).toBe('Rp 185.000');
    expect(formatCurrency(1500000)).toBe('Rp 1.500.000');
  });

  it('returns Rp 0 for zero', () => {
    expect(formatCurrency(0)).toBe('Rp 0');
  });
});

describe('calculateCartTotal', () => {
  it('calculates total correctly for multiple items', () => {
    const cart = [
      { product: { sellPrice: 185000 }, quantity: 2 },
      { product: { sellPrice: 95000 }, quantity: 1 },
    ];
    expect(calculateCartTotal(cart)).toBe(465000);
  });
});
```

### 3.3 Integration Testing

**Framework:** Vitest + Supabase local emulator

Fokus area:
- Server Actions (restockProduct, processTransaction, dll.)
- RLS policies — memastikan kasir tidak bisa mengakses data admin.
- Trigger database — audit log tercatat setelah operasi CRUD.

```typescript
// __tests__/integration/rls.test.ts
```

### 3.4 End-to-End Testing

**Framework:** Playwright

**Alur kritis yang harus di-cover:**

1. **AUTH-E2E-01:** Login sebagai Admin → redirect ke Dashboard.
2. **AUTH-E2E-02:** Login sebagai Kasir → redirect ke POS.
3. **AUTH-E2E-03:** Kasir mencoba akses `/inventory` → redirect ke POS.
4. **POS-E2E-01:** Kasir mencari produk → tambah ke keranjang → pilih channel → proses transaksi → stok berkurang.
5. **INV-E2E-01:** Admin tambah produk baru → produk muncul di daftar inventaris.
6. **INV-E2E-02:** Admin restock produk → stok bertambah sesuai nilai input.

```typescript
// e2e/pos-transaction.spec.ts
import { test, expect } from '@playwright/test';

test('kasir dapat memproses transaksi lengkap', async ({ page }) => {
  await page.goto('/login');
  await page.fill('[data-testid="email"]', 'kasir@maheer.dev');
  await page.fill('[data-testid="password"]', 'Kasir@12345');
  await page.click('[data-testid="login-btn"]');

  await expect(page).toHaveURL('/pos');

  // Cari produk
  await page.fill('[data-testid="product-search"]', 'GM-ALZ');
  await page.click('[data-testid="product-card-0"]');

  // Verifikasi item ada di keranjang
  await expect(page.locator('[data-testid="cart-item-0"]')).toBeVisible();

  // Pilih channel dan proses
  await page.selectOption('[data-testid="channel-select"]', 'offline');
  await page.click('[data-testid="process-transaction"]');

  await expect(page.locator('[data-testid="success-modal"]')).toBeVisible();
});
```

## 4. Deployment Guide

### 4.1 Environment

| Environment | Branch | URL | Supabase Project |
|---|---|---|---|
| Development | `feature/*`, `fix/*` | `localhost:3000` | `maheer-dev` |
| Staging | `develop` | `maheer-staging.vercel.app` | `maheer-staging` |
| Production | `main` | `pos.maheerfashion.com` | `maheer-production` |

### 4.2 Deployment ke Vercel

Deployment dilakukan otomatis via GitHub Actions setiap push ke `develop` atau `main`.

**Manual deploy (jika diperlukan):**

```bash
# Install Vercel CLI
npm i -g vercel

# Deploy ke staging
vercel --env NODE_ENV=staging

# Deploy ke production (wajib konfirmasi eksplisit)
vercel --prod
```

### 4.3 Database Migration

Setiap perubahan schema database harus melalui proses migrasi yang terdokumentasi.

```bash
# Buat file migrasi baru
supabase migration new add_customer_phone_column

# Terapkan migrasi ke database lokal
supabase db push

# Terapkan migrasi ke staging (via Supabase CLI)
supabase db push --db-url $STAGING_DATABASE_URL

# Terapkan ke production (wajib dilakukan saat low-traffic)
supabase db push --db-url $PRODUCTION_DATABASE_URL
```

**Aturan migrasi:**
- Setiap migrasi harus memiliki rollback script di `/supabase/rollbacks/`.
- Migrasi yang mengubah kolom existing atau menghapus tabel wajib melalui approval Admin teknis.
- Migrasi production dilakukan di luar jam operasional (setelah pukul 22:00 WIB).

### 4.4 Pre-deployment Checklist

Sebelum merge ke `main`:

- [ ] Semua environment variables sudah dikonfigurasi di Vercel Production.
- [ ] Database migration sudah diuji di Staging.
- [ ] E2E tests lulus di Staging environment.
- [ ] Tidak ada `console.log` yang tertinggal di kode.
- [ ] Semua secrets sudah dirotasi jika ada perubahan konfigurasi.
- [ ] Backup database production sudah dikonfirmasi tersedia.
- [ ] Tim diberitahu via WhatsApp/Slack tentang jadwal deployment.

### 4.5 Rollback Prosedur

```bash
# Rollback deployment di Vercel
vercel rollback [deployment-url]

# Rollback database migration
supabase db push --db-url $PRODUCTION_DATABASE_URL < supabase/rollbacks/[migration-name]-rollback.sql
```

**Kriteria rollback otomatis:**
- Error rate > 5% dalam 15 menit setelah deployment.
- Response time API > 2 detik rata-rata dalam 10 menit.
- Kegagalan health check endpoint `/api/health`.
